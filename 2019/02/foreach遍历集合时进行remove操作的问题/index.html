增强for循环，即foreach循环其实就是根据list对象创建一个iterator迭代对象，用这个迭代对象来遍历list，相当于list对象中的元素的遍历托管给了iterator，如果对list进行增删操作，都必须经过iterator。

  • iterator.hasNext();判断是否有下个元素
  • item = iterator.next();下个元素是什么，并把它赋给item。

先定位一下该异常的抛出位置，在ArrayList的源码中：

进入ArrayList源码中，可以看到是checkForComodification()方法抛出的异常，也就是说modCount != expectedModCount。

具体原因，是在于使用foreach方式遍历元素的时候，生成iterator，然后使用iterator遍历。在生成iterator的时候，会保存一个expectedModCount参数，这个是生成iterator的时候List中修改元素的次数。如果在遍历过程中删除元素，List中modCount就会变化，如果这个modCount和exceptedModCount不一致，就会抛出异常，这是为了安全考虑。

然而，list中的remove方法并没有对expectedModCount进行任何修改，导致expectedModCount和modCount不一致，抛出异常。

是不是foreach所有的list删除操作都会报出这个错呢

其实不一定，使用上述foreach代码删除的元素是倒数第二个的话，即e4，就不会报错，看一下运行结果：

控制台输出正确，没有任何问题。

之前说了，foreach循环会走两个方法hasNext()和next()。如果不想报错的话，只要不进next()方法就行了，看看hasNext()方法。

如果不进next()方法，就要求hasNext()方法返回false了，即cursor == size。其中cursor是Itr类(Iterator子类)中的一个字段，用来保存当前iterator的位置信息，从0开始。cursor本就是游标的意思。只要cursor不等于size就认为存在元素。由于Itr是ArrayList的内部类，因此直接调用了ArrayList的size字段，所以这个字段的值是动态变化的，既然是动态变化的可能就会有问题出现了。

以上述代码为例，当倒数第二个数据，也就是e4时，cursor是4，然后调用删除操作，此时size由5变成了4，当再调用hasNext()判断时，cursor == size，就会调用后面的操作直接退出循环了，也就不会走后面的异常了。

总结

foreach删除list元素时，只有倒数第二个元素删除不会报错，其他都会报错。

所以，在循环遍历时，还要删除元素最好使用Iterator来遍历元素并进行删除元素。
